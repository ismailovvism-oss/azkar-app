# Азкары — персональное приложение для семьи

> **Основная рабочая версия — публичная: `~/projects/azkar-app`
> (репозиторий `azkar-app`).** Все правки контента и интерфейса по
> умолчанию делаются там. Семейная версия (`~/projects/azkary`,
> репозиторий `azkary`) синхронизируется из публичной **только по
> явной просьбе** автора — скриптом `~/projects/azkar-app/sync-family.sh`.
> Подробнее — в разделе «Деплой».

## О проекте

Подарок жене и четырём детям на Курбан-байрам. Один HTML-каркас
(`azkary.html`), один общий `content.json` с зикрами и общими
комментариями автора, плюс маленькие персональные файлы
`personal/*.json` с именем, фото, личным письмом и личными
комментариями к избранным зикрам.

Ценность подарка — не в технике, а в этих личных посланиях
(`letter`, `personalComments`). Приложение — обёртка для них.

## Аудитория

Жена (взрослая) и четверо детей разного возраста. Тон —
тёплый и домашний, но изящный и сдержанный.

## Архитектура

**Три слоя:**

1. **Код** — `azkary.html`. Vanilla JS, без сборщиков и фреймворков,
   всё в одном файле. Стиль кода как в проекте math-trainer (один
   глобальный `state`, функция `render()` по `state.screen`,
   утилиты `lsGet`/`lsSet`/`esc`). **Идентичен** в семейной и
   публичной версиях.
2. **Общий контент** — `content.json`. Только массив `sections` с
   зикрами. Поле `items[].comment` — общий комментарий к зикру,
   виден всем пользователям. Никаких личных данных тут нет.
3. **Личное** — папка `personal/`, по одному JSON-файлу на
   получателя (`wife.json`, `hamdiya.json`, …). Содержит имя, фото,
   личное письмо (`letter`), и словарь `personalComments` —
   индивидуальные комментарии к конкретным зикрам, видимые только
   этому одному человеку.

**Выбор учётки — URL-параметром `?u=name`.** Без параметра
приложение пробует загрузить `personal/default.json`. В семейной
сборке этого файла нет — поэтому без `?u=` приложение покажет
экран ошибки, а не «дефолтную» учётку. Это сознательная защита:
ссылка без имени никому не нужна, личное должно быть адресным.

**Два режима** (`mode` в `personal/*.json`):
- `family` — закрытая семейная сборка. Имя приходит из файла, его
  нельзя редактировать в приложении. Файла `personal/default.json`
  нет, без `?u=` — ошибка.
- `public` — общедоступная сборка. `personal/default.json` есть,
  имя в нём обычно пустое — приложение спросит у пользователя при
  первом запуске и запомнит в `localStorage` (`azkary.publicUserName`).

`mode` — обязательное поле в `personal/*.json`. Никаких других
значений быть не должно.

**Публичная версия — особенности UI** (всё под `mode: public`, код
единый, семейная версия этих элементов не показывает):
- предисловие (`letter`) показывается **вверху** главного экрана над
  разделами; длинное (>220 символов) сворачивается «Читать дальше»;
- экран **«Как пользоваться»** (справка о механике приложения, текст
  захардкожен — одинаков для всех): вход ссылкой под предисловием и
  пунктом ⚙ Настройки → Справка;
- если задан `feedbackEmail` — внизу главной кнопка **«Предложи свой
  зикр или функцию»**, ведущая на экран обратной связи (адрес текстом
  + «Написать письмо» mailto + «Скопировать адрес»).

**Защита публичной версии от подмены.** Если параметр `?u=имя`
указан, но загруженный `personal/{имя}.json` оказался `mode:
public` — приложение принудительно перезагружает
`personal/default.json`. Это блокирует попытку «подобрать» имя
семейной учётки на публичной сборке.

**Хранилище — `localStorage`.** Прогресс/заметки/кастомные повторы.
Адресация — по стабильному `item.id` из `content.json`, не по
позиции. Slug в ключах LS вычисляется через `getUserSlug()` →
`getUserName()`. Полный список префиксов — в разделе «Хранилище
и сброс данных».

**Имя пользователя — только через хелперы.** Все обращения к имени
получателя идут через `getUserName()`, slug для ключей localStorage —
через `getUserSlug()`. **Никаких прямых `state.personal.user.name`
или localStorage-чтений ключа `publicUserName`** в коде вне этих
хелперов.

**Версия схемы данных** — `SCHEMA_VERSION = 2`. Меняется только при
структурной миграции ключей localStorage. Текущая миграция (с
(sectionId, idx) на zikrId) — единственная активная. Любая будущая
такая миграция обязана соблюдать паттерн `preMigrationBackup.vN`
(см. раздел «Хранилище и сброс данных»).

**Распространение: Netlify Drop.** Семейная версия — один сайт на
всю семью; каждому члену семьи отдаётся ссылка с его параметром
`?u=…`. Публичная версия — отдельный сайт, отдельная папка. Код
`azkary.html` в обеих версиях идентичен, различие — только в
наполнении `personal/`. Подробности — в `README.md`.

## Формат `content.json` (общий)

```json
{
  "sections": [
    {
      "id": "morning",
      "title": "Утренние азкары",
      "subtitle": "После рассвета до восхода",
      "items": [
        {
          "id": "morning-001",
          "title": "Короткий понятный заголовок",
          "arabic": "أَعُوذُ بِاللَّهِ مِنَ الشَّيْطَانِ الرَّجِيمِ",
          "transliteration": "А‘узу билляхи мина-ш-шайтани-р-раджим",
          "translation": "Прибегаю к Аллаху от проклятого шайтана",
          "repeat": 3,
          "source": "Сура «Ан-Нахль», 16:98",
          "tags": ["утро", "защита"],
          "comment": "Общий комментарий к зикру — тафсир/шарх — виден всем"
        }
      ]
    }
  ]
}
```

В `content.json` **нет** полей `user`, `author`, `letterFromDad` —
всё личное переехало в `personal/*.json`. Валидатор молча
игнорирует эти поля, если они случайно остались от старого
формата, но код их не использует.

### Обязательные поля content.json

- `sections[]` — массив разделов. Порядок определяет порядок
  карточек на главном экране.
- `sections[].id` — короткий стабильный идентификатор (`morning`,
  `evening`, `sleep`, …). Используется в hash-роутинге; в ключах
  localStorage НЕ используется (адресация по `item.id`). Менять
  без причины не стоит.
- `sections[].title` — заголовок раздела.
- `sections[].items[]` — массив зикров. Порядок определяет порядок
  прохождения.
- `items[].id` — **уникальный** идентификатор зикра. Рекомендуемый
  формат `{sectionId}-NNN` (`morning-001`, `evening-012`).
  Уникальность проверяется по всему файлу. По этому id адресуются
  ВСЕ пользовательские данные (заметки, прогресс, кастомные
  повторы) **и** личные комментарии в `personal/*.json`. Изменение
  `id` существующего зикра приведёт к архивации его LS-данных и
  «потере» личных комментариев у всех — для всех ключ `personalComments[id]`
  перестанет совпадать с актуальным id.
- `arabic`, `transliteration`, `translation`, `repeat` — у каждого
  зикра обязательны.

### Опциональные поля content.json

- `sections[].subtitle` — короткий подзаголовок раздела.
- `items[].title` — короткий понятный заголовок конкретного зикра.
  Показывается в списке зикров и на экране зикра. Особенно полезен
  для разделов с ситуативными дуа: «Перед едой», «Выход из дома».
- `items[].source` — короткая ссылка на источник («Бухари 6310»,
  «Сура „Ан-Нахль", 16:98»). Показывается под переводом.
- `items[].tags` — массив строк-меток. Рендерятся как «пилюли» под
  source-строкой. Фильтрация по тегам — пока не сделана.
- `items[].comment` — **общий** комментарий к зикру (тафсир/шарх),
  виден всем пользователям. Отдельная карточка с заголовком
  «Комментарий». Длинные тексты (>200 символов) автоматически
  сворачиваются.

## Стандарты контента (source и comment)

Этот раздел — обязательная инструкция для всех, кто добавляет
или редактирует зикры в content.json. Соблюдается всегда, не
"по возможности". Если автор просит добавить новый зикр без
указания source и comment — сначала предложи их по этим
стандартам, потом добавляй.

### Поле source

Указывает первоисточник зикра. Формат и содержание зависят от
типа источника.

**Если зикр из Корана** — указывается название суры и номер
аята: «Сура аль-Касас, аят 24». Если несколько аятов подряд —
диапазон: «аяты 13-14». Транслитерация названия суры на
русский в стиле, принятом в проекте (аль-Бакара, ан-Ниса,
аль-Касас).

**Если зикр из Сунны** — указывается передатчик и сборник
(сборники, если их несколько). Формат: «Хадис от [имя
сахаба], у имама [имя имама-составителя]».

Примеры:
- «Хадис от Зейда ибн Халида аль-Джухани, у имамов аль-Бухари
и Муслима»
- «Хадис от ибн Умара, у имама Абу Дауда»
- «Хадис от Абу Хурайры, у имамов ат-Тирмизи и ан-Насаи»

**Если в источнике несколько хадисов или элементов** (например,
кораническая основа плюс хадис о применении) — оба указываются
через точку или связку: «Сура аз-Зухруф, аяты 13-14. В Сунне
передаётся, что Пророк ﷺ произносил эту формулу при поездке
на верховом животном».

**Если источник содержит дополнительные классические тафсиры**
(используемые в comment) — они тоже указываются в source:
«Тафсир Ибн Аббаса (да будет доволен им Аллах) к аяту
аль-Бакара 22 — у имама Ибн Аби Хатима».

### Степень достоверности иснада

Указывается **только если есть необходимость** — когда хадис
не из двух Сахихов и есть учёная дискуссия о его достоверности,
или когда автор контента считает важным указать степень.

**Используются только классические термины** (без современной
терминологии и без оценок современных учёных):
- сахих (достоверный)
- хасан (хороший)
- даиф (слабый)
- мауду (вымышленный — встречается редко)

Формат: «Хасан хадис от ... у имама ...» или «Иснад слабый,
у имама ...».

**Если хадис из аль-Бухари или Муслима** — достоверность не
указывается отдельно, потому что нахождение в этих сборниках
само по себе подтверждает её для подавляющего большинства
случаев.

**Если есть учёное разногласие о достоверности** — указывается
обтекаемой формулировкой: «достоверность обсуждается»,
«некоторые мухаддисы считали хасан, некоторые усиливали до
сахих». Без приписывания этих оценок конкретным современникам.

**НИКОГДА не упоминаются современные учёные** (XIX-XXI веков)
по именам или их работам. Это включает аль-Албани, ибн База,
ибн Усеймина, аль-Кахтани (Хисн аль-Муслим), Darussalam, и
им подобных. Если в источнике есть оценка иснада, исходящая
от современников — она передаётся без атрибуции, в страдательном
залоге: «иснад слабый», «достоверность спорная», без указания
кем именно оценено.

### Поле comment (шарх)

Это **главный обучающий элемент** карточки зикра. Не просто
«перевод другими словами», а раскрытие смысла, контекста,
применения и эмоционального содержания зикра.

**Длина и тон.** Шарх — содержательный, развёрнутый. Не
3 предложения, не 5 абзацев формальности. Объём — столько,
сколько нужно для раскрытия смысла. Обычно 4-7 абзацев. Тон
тёплый, личный, обращённый к читателю на «ты». Не учебниковый,
не сухой, не как фетва. Скорее как разговор взрослого с
взрослым о важном.

**Структура шарха.** Не жёсткая, но в большинстве случаев
содержит следующие элементы (в свободном порядке, какие
нужны):

1. **Контекст возникновения.** Если это хадис — где и при
каких обстоятельствах был сказан. Если это аят — какая
история, чьи слова, какая ситуация в суре. Без перегрузки
деталями — только то, что помогает понять смысл.

2. **Раскрытие смысла.** Не пересказ перевода, а **то, что
стоит за словами**. Часто это требует обратить внимание на
конкретное слово или конструкцию в арабском: его этимологию,
его коннотации, его связь с другими аятами или хадисами.
Цель — чтобы читатель увидел в зикре глубину, которой не
видел до сих пор.

3. **Связь с общей картиной ислама.** Как этот зикр связан
с другими известными хадисами, аятами, принципами. Иногда —
указание на аналогичные дуа или формулы, выражающие ту же
идею в другом ракурсе. Цель — показать, что зикр не
изолированный текст, а часть **системы**.

4. **Практическое применение.** Когда произносить, в каком
состоянии сердца, в каких ситуациях. Если зикр имеет узкий
контекст в сунне, но широкий смысл — указать на это (как
в случае с саххара, переданным в контексте поездки, но
выражающим общий принцип).

5. **Эмоциональный отклик.** Хороший шарх не просто объясняет
смысл, но и **поднимает** уместное переживание: благоговение,
благодарность, страх Аллаха, любовь к Нему, надежду. Не
искусственно навязывается, а возникает естественно из
раскрытого смысла.

**Чего избегать в шархе.**

- Не превращать в проповедь с восклицаниями («Подумай только!»,
«Какая красота!»). Эмоция передаётся через **сам смысл**,
не через эмоциональную накачку.
- Не цитировать современные книги тафсира или комментаторов
XX-XXI веков. Опираться на классические тафсиры (ат-Табари,
Ибн Касир, аль-Куртуби, аль-Багави, ибн Аби Хатим) и
классических учёных (ибн Таймия, ибн аль-Каййим, ибн Раджаб,
ан-Навави и ранее).
- Не выносить фетв и не делать категоричных утверждений по
спорным фикховым вопросам. Если есть разногласие — упомянуть
обе позиции, без выбора «правильной».
- Не использовать арабские термины без перевода на русский
при первом упоминании. Если термин ключевой и встречается
часто — дать русское объяснение в скобках: «куфр ан-ни'ма
(неверие в милость)».
- Не превращать шарх в развёрнутый учебник по теме. Если
тема большая — обозначить её, дать суть, и дать читателю
самому углубиться при желании.

### Использование классических тафсиров в шархе

Когда в шархе цитируется классический тафсир — атрибутируется
конкретно: «Ибн Аббас (да будет доволен им Аллах) сказал... —
этот тафсир передаёт имам Ибн Аби Хатим». Без указания тома
и страницы в тексте шарха (это перегружает). Если автор
контента предоставил точную ссылку (книга, том, страница,
номер передачи) — можно вынести её в source как
дополнительный элемент, но в самом comment оставлять
читаемый стиль.

### Уважительные формулы

- После упоминания Пророка Мухаммада — ﷺ (или «мир ему и
благословение Аллаха»). Сокращение «ﷺ» допустимо как более
компактное.
- После упоминания других пророков — «(мир ему)» или «(мир
ему и благословение Аллаха)».
- После упоминания сахаба или сахабийки — «(да будет
доволен им/ею Аллах)».
- После упоминания учёного из числа классиков — «(да
помилует его Аллах)» — необязательно, по контексту.

### Пример хорошего шарха (как ориентир)

См. шархи к зикрам в разделе "За все милости" в текущем
content.json — особенно к зикру «Мутирна би фадлилляхи»,
к «Субханаллази саххара», к «Дуа Мусы». Они выдержаны в
стиле, который и есть стандарт проекта.

### Поле title

Короткий заголовок зикра, отображаемый в карточке списка
зикров. Не дословный перевод, а **смысловая зацепка**,
выражающая суть зикра одной фразой.

**Длина:** в среднем 3-7 слов. Должен помещаться в карточку
без переноса на вторую строку (ориентировочно до 40-45
символов).

**Стиль:** короткая фраза от первого лица или фраза-принцип.
Не сухой ярлык, не "техническое название". Скорее как
голос внутри головы человека, который произносит этот зикр.

Примеры из проекта:
- «В каждом благе от Тебя нуждаюсь» (дуа Мусы)
- «А сами бы мы это не смогли» (саххара)
- «По милости Аллаха, не от моих заслуг» (мутирна)

### Поле tags

Свободные теги для тематической группировки. Используется
для будущих фильтров и подборок. Несколько коротких слов
на русском. Примеры: «благодарность», «милость», «защита»,
«дорога», «утро», «вечер», «таухид», «надежда».

Один зикр обычно имеет 2-4 тега. Слишком много тегов —
теряется ценность фильтрации.

### Поле repeat

Количество рекомендуемых повторений.

- Для аятов Корана как зикров — 1 (если в сунне не передано
иного количества).
- Для коротких формул-тасбих (Субханаллах, Альхамдулиллях,
Аллаху Акбар, Ля иляха илля-Ллах) — обычно 33 или 100
по сунне.
- Для дуа (просьбы к Аллаху) — обычно 1, произносится
осознанно один раз.
- Если есть конкретное число из сунны — использовать его
с указанием в source/comment.

### Финальная проверка перед сохранением

Перед коммитом нового зикра автор (или AI) проверяет:

1. У зикра есть все обязательные поля: id, arabic,
transliteration, translation, repeat, title.
2. У зикра есть source — указаны корректные источники по
формату выше.
3. У зикра есть comment — соответствует стандартам по длине,
структуре, тону.
4. Нет упоминаний современных учёных в source и comment.
5. Уважительные формулы расставлены везде где надо.
6. JSON валидный, ID уникальный.
7. Арабский текст — стандартный, с огласовками, корректный.

### Что делать при сомнении

Если у автора контента возникает сомнение по любому пункту
(достоверность хадиса, корректность тафсира, методологическая
позиция) — **сомнение приоритетнее быстроты**. Лучше
оставить плейсхолдер «уточнить источник» и спросить автора
лично, чем выдать что-то, в чём AI не уверен. Особенно
это касается номеров хадисов в сборниках и атрибуции цитат
конкретным учёным.

## Формат `personal/{name}.json` (личное)

### Семейная учётка (`mode: family`)

```json
{
  "mode": "family",
  "user": {
    "name": "Ханифа",
    "photo": "photos/hanifa.jpg"
  },
  "author": {
    "name": "папы",
    "photo": "photos/papa.jpg"
  },
  "letter": "Моя дорогая Ханифа, ...",
  "personalComments": {
    "morning-001": "Тебе лично к этому зикру...",
    "evening-003": "..."
  }
}
```

### Публичная учётка (`mode: public`)

Лежит в `personal/default.json` в публичной сборке. В семейной
сборке этого файла нет.

```json
{
  "mode": "public",
  "user": { "name": "", "photo": "" },
  "author": { "name": "автора", "photo": "" },
  "letter": "Общее вступление для всех читателей...",
  "feedbackEmail": "qaadiy@gmail.com",
  "personalComments": {}
}
```

В публичной версии фото обычно нет (`photo` пустые) — код их и так
прячет. `author.name` в публичке не показывается (ссылки «Слово от …»
нет — предисловие вынесено вверх, см. ниже).

### Поля personal/*.json

- `mode` — **обязательное**, только `"family"` или `"public"`.
- `user` — объект. В режиме `family` поле `user.name` —
  обязательное, непустое. В режиме `public` имя может быть пустым
  (тогда приложение спросит при первом запуске).
- `user.photo` — путь к фото получателя (например,
  `"photos/wife.jpg"`). По соглашению — относительно корня сайта,
  файл лежит в `photos/`. Если файла нет — экран рендерится без
  фото, без сообщения об ошибке.
- `author` — объект, опциональный целиком. Внутри тоже всё опционально:
  - `author.name` — имя автора подарка **в родительном падеже**.
    Появляется только в ссылке «✉ Слово от …» на главном экране
    (`"мужа"` → «Слово от мужа», `"папы"` → «Слово от папы»). Если
    не указано — «Слово от автора». В подписи письма больше не
    используется: автор подписывается сам внутри текста письма.
  - `author.photo` — путь к фото автора. Показывается рядом с
    подписью в письме.
- `letter` — личное письмо. Если поле отсутствует или пустая
  строка — кнопка «✉ Слово от …» с главного экрана не показывается
  вообще (исключение: на экране ввода имени в публичном режиме
  письмо отображается как общее вступление). Поддерживает
  переносы строк (`\n`) и мини-markdown (`**жирный**`, `*курсив*`).
  **Поле бывшего `letterFromDad` теперь называется `letter`.**
  В режиме `public` `letter` работает как предисловие автора и
  показывается **вверху** главного экрана над разделами (длинное —
  сворачивается), а не за кнопкой внизу.
- `feedbackEmail` — email для обратной связи (опциональный, строка).
  Если задан, внизу главного экрана появляется кнопка «✎ Предложи
  свой зикр или функцию» → экран обратной связи: адрес показан
  текстом (буквами, можно выделить), кнопки «Написать письмо»
  (mailto с готовой темой и заготовкой) и «Скопировать адрес»
  (Clipboard API + запасное выделение). Сознательно не «слепой»
  mailto: почтовый клиент может не открыться, поэтому адрес всегда
  виден. По смыслу — поле публичной версии; в семейных файлах не
  задаётся (тогда кнопки нет).
- `personalComments` — словарь `{ zikrId: текст }`. Опциональный,
  по умолчанию пустой `{}`. На экране зикра, чей id есть в этом
  словаре, появляется карточка «💌 Тебе лично» под общей карточкой
  «Комментарий». Только этот один пользователь видит эту карточку.
  Длинные тексты (>200 символов) автоматически сворачиваются.

## Экран зикра

### Зона счёта (выбор активной зоны тапа)

На длинных дуа арабский текст оказывается вверху экрана, а круглая
кнопка-счётчик уходит вниз — читать и тапать одновременно неудобно.
Поэтому пользователь может выбрать, **по какой области экрана**
засчитывается повтор. Настройка — в ⚙ Настройки → группа «Счётчик»
(отдельно от «Оформления»: это поведение, не вид).

Пять вариантов (radio-выбор, активен один). Источник истины — массив
`TAP_ZONE_OPTIONS` в `azkary.html`:

- **`circle`** (по умолчанию) — текущее поведение, тап по круглой кнопке
  внизу. Никаких рамок и бейджей.
- **`arabic`** — тап по блоку арабского текста.
- **`translit`** — тап по блоку транслитерации.
- **`translation`** — тап по блоку перевода.
- **`screen`** — тап по любому месту экрана зикра.

**Общий принцип (круг-табло + бейдж).** Круглая кнопка-счётчик внизу
остаётся ВСЕГДА на месте как главное «табло» и сама всегда кликабельна
(тоже считает), независимо от выбранной зоны. Для текстовой зоны
дополнительно: вокруг соответствующего блока — тонкая полупрозрачная
охровая рамка-индикатор (`.tap-zone-active`, рамка через `::after` с
`opacity`), а в правом верхнем углу — бейдж-дубль счётчика
(`.zone-badge`, `id="zone-badge"`). Бейдж и круг держатся в синхроне
через `updateTapperDom()` (одно число, одно состояние «Готово»). Для
`screen` рамок и бейджа нет — активна вся область, табло только круг.

**Скролл vs тап.** Для текстовых зон и `screen` тап отличается от
прокрутки: если палец между нажатием и отпусканием сдвинулся больше
`TAP_MOVE_TOLERANCE` (~10px) — это скролл, повтор не засчитывается
(`attachTapCounter`, Pointer Events + `pointercancel`). Длинные дуа
прокручиваются и не должны накручивать счётчик.

**Исключения.** Во всех зонах, кроме круга, тап по интерактивному
элементу не засчитывает повтор: кнопки (Назад, Обнулить, ✎, ↺, навигация),
теги, ссылки, поле заметок. Селектор — `TAP_EXCLUDE_SELECTOR`
(`button, a, textarea, input, select, label, [role="button"], .tag-pill,
.user-notes`), проверка через `closest()`.

**Вибрация** при засчитанном повторе сохраняется для всех зон
(`navigator.vibrate`, как у круга). Поведение при достижении `repeat`
(подсветка «Готово», автопереход) не меняется.

Зона — настройка **устройства** (slug-независимый ключ `azkary.tapZone`,
по аналогии с `azkary.theme`), общая для всех учёток в браузере, доступна
в обоих режимах. По умолчанию — `circle`: текстовые зоны срабатывают
только в своих границах, «весь экран» — сознательно НЕ дефолт (бывают
случайные касания экрана, например чтобы он не гас).

## Хранилище и сброс данных

В localStorage живут **три независимых типа** пользовательских данных
плюс служебные ключи. Все адресуются по уникальному `item.id`
(а не по позиции зикра в content.json). Префикс ключей включает
slug пользователя — `getUserSlug()`.

| Префикс ключа                                  | Что                                      | Срок жизни            | UI-инструмент сброса                  |
| ---------------------------------------------- | ---------------------------------------- | --------------------- | ------------------------------------- |
| `azkary.{slug}.progress.{date}.{zikrId}`       | счётчики тапов                            | один день (автосброс) | ↺ на экране зикра, «Обнулить» в шапке раздела, «Обнулить весь прогресс на сегодня» в ⚙ |
| `azkary.{slug}.note.{zikrId}`                  | заметки пользователя                      | бессрочно             | «Удалить все мои заметки» в ⚙ |
| `azkary.{slug}.customRepeat.{zikrId}`          | кастомные повторы                         | бессрочно             | «Сбросить к рекомендуемому» в пикере на экране зикра, «Сбросить все кастомные повторы» в ⚙ |
| `azkary.{slug}.archive`                        | архив осиротевших данных (массив)         | бессрочно             | «Удалить архив» в ⚙ → Архив |
| `azkary.{slug}.contentSnapshot`                | карта `{zikrId → {transliteration, sectionTitle}}` для будущих архивирований | перезаписывается на каждом старте | (служебный) |
| `azkary.{slug}.schemaVersion`                  | текущая версия схемы (`SCHEMA_VERSION = 2`) | бессрочно             | (служебный) |
| `azkary.{slug}.preMigrationBackup.v{N}`        | снапшот всех ключей **до** миграции        | 30 дней (автоудаление) | (служебный, чистится `cleanupStaleMigrationBackups`) |
| `azkary.{slug}.lastMigrationReport.v{N}`       | счётчики последней миграции                 | до закрытия плашки    | (служебный) |
| `azkary.{slug}.migrationNoticeShown.v{N}`      | флаг «пользователь увидел отчёт»            | бессрочно             | (служебный) |
| `azkary.publicUserName`                        | имя, введённое в публичном режиме           | бессрочно             | «Изменить имя» в ⚙ → Профиль (только в public) |
| `azkary.theme`                                 | выбранная цветовая схема (`cream`/`dawn`/`twilight`/`night`) | бессрочно | «Оформление» в ⚙ |
| `azkary.tapZone`                               | зона тапа счётчика на экране зикра (`circle`/`arabic`/`translit`/`translation`/`screen`) | бессрочно | «Счётчик» в ⚙ |

`azkary.publicUserName`, `azkary.theme` и `azkary.tapZone` — **slug-независимые** ключи.
`publicUserName` читается и пишется только в публичном режиме; задаёт
имя, из которого далее вычисляется slug для всех остальных ключей этого
пользователя. При смене имени через «Изменить имя» приложение пытается
перенести все ключи под новый slug, чтобы заметки/прогресс/настройки не
потерялись.

`azkary.theme` — настройка оформления **устройства**, общая для всех
учёток в этом браузере (не привязана к получателю). Источник истины по
списку схем, цвету системной шапки (`<meta theme-color>`) и фону
PWA-манифеста — массив `THEMES` в `azkary.html`. Палитра каждой схемы —
переменные CSS под `html[data-theme=…]`; выбор применяется через
`applyTheme()` ещё до первого `render()` (чтобы не мелькал дефолтный фон).
Схема `cream` — дефолтная (без атрибута `data-theme`). При добавлении
новой схемы правится `THEMES` (id/label/bg/swatch) **и** парный
CSS-блок `html[data-theme="<id>"]`. Все схемы держатся одного принципа —
тёплая, бумажная, сдержанная; см. «Эстетические принципы».

`azkary.tapZone` — настройка поведения счётчика **устройства** (общая для
всех учёток браузера). Определяет, по какой области экрана зикра
засчитывается повтор; подробно — в разделе «Экран зикра → Зона счёта».
Источник истины по списку зон — массив `TAP_ZONE_OPTIONS` в `azkary.html`.

### Принципы

**1. Раздельный сброс.** Счётчики, заметки и кастомные настройки повторов
обнуляются независимо. Объединённого «сброса всего», кроме отдельной
кнопки «Удалить всё для этого пользователя» с двойным подтверждением,
**нет**. Если возникнет соблазн добавить «reset all» — стоп, подумай,
какие именно данные пользователь хочет потерять, и сделай отдельную
кнопку.

**2. Стабильные ID.** Пользовательские данные привязаны к
`item.id`, а не к позиции зикра. Перестановка зикров и
переименование текста — безопасны, заметки остаются у того же
зикра. Изменение самого `id` или удаление зикра из content.json
— это переезд LS-данных в архив, не потеря. Личные комментарии
(`personalComments`) тоже привязаны к `id` — при его изменении
комментарий «теряется» (молча: блок «Тебе лично» просто
перестаёт показываться).

**3. Архив вместо удаления.** При обнаружении localStorage-ключа
с `zikrId`, которого больше нет в content.json, его содержимое
переезжает в `archive` (с последним известным
`transliteration`/`sectionTitle` из `contentSnapshot`). Архив
видно через ⚙ → Архив, можно скачать TXT или удалить отдельной
кнопкой. **Автоматически архивные записи не удаляются никогда.**
(Личные комментарии в архив НЕ переезжают — они в файле, а не в LS.)

**4. Паттерн миграции** (для будущих структурных миграций):

Любая будущая миграция, меняющая формат ключей localStorage,
**обязана**:
1. **Записать снапшот** всех ключей пользователя в
   `azkary.{slug}.preMigrationBackup.v{N}` (с `createdAt`) **до**
   любых изменений. Если запись провалилась (квота, приватный
   режим Safari и т.п.) — миграцию НЕ запускать, показать
   error-экран. Лучше явная ошибка, чем потерянные данные.
2. **Накопить отчёт** `{ migrated*, archived* }` и записать как
   `azkary.{slug}.lastMigrationReport.v{N}`.
3. **Показать пользователю одноразовое уведомление** на home с
   цифрами из отчёта. Флаг просмотра —
   `azkary.{slug}.migrationNoticeShown.v{N}`. Снимается крестиком
   или первым открытием архива.
4. `preMigrationBackup.v{N}` автоматически удаляется через 30
   дней через `cleanupStaleMigrationBackups()`.

Это даёт страховку при сбое (всегда есть бэкап в localStorage,
доступный через DevTools) и снимает тревогу у пользователя —
он видит конкретные цифры, а не пустоту вместо своих заметок.

## Загрузка и валидация при старте

Порядок шагов в `init()`:

1. Парсим URL `?u=name`; sanitize до латиницы/цифр/дефиса. Если
   параметра нет — берём `default`.
2. Загружаем `personal/{name}.json`. 404 → экран ошибки с разной
   формулировкой в зависимости от того, был ли `?u=` указан.
3. Валидируем `personal` (`validatePersonal`). Проверяет `mode`
   и обязательные/опциональные поля для каждого режима.
4. **Защита публичной версии.** Если `mode: public` И `?u=` был
   указан с не-`default` значением — перезагружаем
   `personal/default.json`.
5. Загружаем `content.json`. Валидируем (`validateContent`) —
   только `sections`.
6. **Публичный режим без имени.** Если `mode: public` и
   `user.name` пуст, и в LS нет `azkary.publicUserName` —
   показываем экран ввода имени (`enterName`). Миграцию НЕ
   запускаем (slug ещё не определён). После ввода имени —
   продолжаем с шага 7.
7. Чистка просроченных миграционных бэкапов
   (`cleanupStaleMigrationBackups`).
8. `maybeMigrate(report)` — миграция LS-ключей со старого формата
   на новый, если ещё не была сделана.
9. `processOrphans(report)` — поиск осиротевших LS-ключей и
   переезд в архив.
10. Запись отчёта миграции в LS (для одноразовой плашки на home).
11. `cleanupOldProgress()` — удаление вчерашнего прогресса.
12. `applyHash()` + слушатели + `render()`.

## Типичные ошибки JSON

Если приложение показывает экран ошибки — почти всегда виновата одна
из этих опечаток в `content.json` или `personal/*.json`:

- **Запятая после последнего элемента** массива или объекта. JSON
  такого не прощает: `[1, 2, 3,]` или `{ "a": 1, }` — синтаксическая
  ошибка.
- **«Умные» кавычки** вместо обычных. При копировании текста из
  Word/Pages/Telegram кавычки могут стать `"…"` или `«…»` вместо `"…"`.
  Все строковые значения должны быть в *прямых* двойных кавычках.
- **Пропущенная запятая** между элементами объекта или массива.
- **Незакрытая скобка** — `{`, `[` или строка без закрывающей `"`.
- **Лишний BOM** в начале файла при сохранении из Notepad — JSON-парсер
  ругается. Сохраняй как «UTF-8 без BOM».
- **Опечатка в режиме**: `"mode": "famly"` или `"mode": "private"`.
  Допустимы только `"family"` и `"public"`.

Лучший редактор для правок — VS Code: он сразу подсвечивает
синтаксические ошибки красной волной.

## Деплой

**Семейная и публичная версии — две отдельные папки, два репозитория
GitHub и два отдельных сайта Netlify** (Netlify тянет из GitHub):

| | Семейная | Публичная |
|---|---|---|
| Папка | `~/projects/azkary` | `~/projects/azkar-app` |
| GitHub | `azkary` (приватный) | `azkar-app` (публичный) |
| `personal/` | `wife`, `hamdiya`, `hammad`, `hanifa`, `ayat` (все `mode: family`) | только `default.json` (`mode: public`) |

Код `azkary.html` и общий `content.json` в обеих папках **идентичны**.
Различие — только в составе `personal/`. В семейной папке `default.json`
лежит локально для теста, но он в `.gitignore` — на сайт не попадает
(без `?u=` там обязан быть экран ошибки).

**Публичная версия (`azkar-app`) — основная рабочая.** Все правки
контента и интерфейса по умолчанию делаются в `~/projects/azkar-app`.
Семейная версия синхронизируется из публичной **только по явной
просьбе** автора.

`azkary.html` и `content.json` синхронизируются **вручную, из
публичной в семейную**, скриптом, который лежит в публичной папке:

```
~/projects/azkar-app/sync-family.sh
```

Он копирует `azkary.html` и `content.json` из `azkar-app` → `azkary`.
`personal/` не трогается (у версий разные учётки). После синхронизации
оба репозитория нужно закоммитить и запушить — Netlify пересоберёт
сайты сам.

## Инструкция для Claude Code в будущих сессиях

Когда родитель просит:

- **Добавить новый зикр или раздел** — добавляй в `content.json`
  строго по приведённой выше схеме (отступы 2 пробела, порядок
  ключей: `id`, `arabic`, `transliteration`, `translation`,
  `repeat`, затем опциональные `source`, `tags`, `comment`).
- **Написать общий комментарий к зикру** — это `items[].comment` в
  `content.json`, виден всем.
- **Написать персональный комментарий конкретному человеку** — это
  `personalComments[zikrId]` в его файле в `personal/`. Спроси, к
  какому именно зикру (по id или по содержимому).
- **Добавить нового члена семьи** — создай новый файл в `personal/`
  по шаблону семейной учётки. Фото клади в `photos/`. Дай родителю
  готовую ссылку с правильным `?u=`.
- **Перенести содержимое из `personal/X.json` в `personal/Y.json`**
  (например, при ошибке) — действуй осторожно, убедись, что не
  затрёшь существующие данные в Y.

Важные напоминания:
- **Поле `letterFromDad` из прошлой версии полностью заменено на
  `letter`** (в `personal/*.json`).
- **Поле `note` (ещё более старое) заменено на `comment`** (в
  `content.json`).
- **`comment` в content.json и `personalComments[id]` в
  personal/*.json — РАЗНЫЕ ВЕЩИ.** Первое видно всем, второе —
  только одному человеку.
- **Имя получателя — только через `getUserName()`.** Никаких
  прямых обращений к `state.personal.user.name` в коде вне самой
  функции `getUserName()`.
- Арабский текст — справа налево, **не переворачивай руками**:
  храним логический порядок символов, направление контролирует CSS
  (`dir="rtl"`) на стороне отображения.
- После правки покажи родителю **только добавленный/изменённый
  фрагмент**, не весь файл целиком. Файлы вырастут, и каждый раз
  печатать их — шумно.
- Если родитель просит «добавить транслитерацию к арабскому» или
  «перевод» — спроси, какой стиль он предпочитает (например, для
  транслитерации: с диакритиками `ṣ`, `ḥ` или без; стиль казанский /
  стандартный международный) и сохрани его ответ в этот файл, чтобы
  следующие зикры были в одном стиле.

## Эстетические принципы

- Тёплая и домашняя атмосфера, но изящная и сдержанная. Бумага, тихий
  вечер, минимум границ, много воздуха.
- Палитра — кремовая (`--bg: #F5EFE3`), охра как акцент (`--accent:
  #8B5E34`), золото для подписей (`--gold: #B5895A`). Это схема `cream`
  по умолчанию. Доступны и другие цветовые схемы (⚙ → «Оформление»):
  `dawn` (тёплая заря, пыльная роза), `twilight` (приглушённый вечерний
  мововый тон), `night` (тёплая тёмная бумага для чтения в темноте).
  **Любая новая схема обязана держаться этих же принципов** — тёплая,
  бумажная, сдержанная, без кича; добавлять схему = новый элемент в
  `THEMES` + парный CSS-блок `html[data-theme=…]` (см. «Хранилище»).
- Серифные шрифты: Amiri (классическая naskh) для арабского,
  Cormorant Garamond для дисплейных надписей на кириллице/латинице,
  DM Sans для UI-надписей.
- **Никакого «исламского» кича**: зелёные мечети, золотые орнаменты,
  крупные звёзды и полумесяцы — запрещены. Декоративные элементы —
  только тонкие линии и точки-разделители (`·`).
- Эмодзи в UI почти отсутствуют. Допустимые исключения — ✉ для
  ссылки на письмо, 📝 как индикатор личной заметки в списке зикров,
  💌 как индикатор личного комментария «Тебе лично» на экране зикра,
  ⚙ для входа в настройки.
- Без градиентов, без резких теней, без ярких акцентов.

Если возник вопрос «как сделать красивее» — выбирай **сдержаннее**, не
ярче.

## Roadmap

**Сделано:**
- Каркас приложения (главный экран, экран раздела, экран зикра, письмо,
  экран ошибки, настройки).
- Базовый контент для жены: 5 разделов из Хисн аль-муслим, по 3–5
  зикров в каждом.
- Счётчик-таппер с вибрацией и автопереходом.
- Дневной сброс прогресса.
- PWA: manifest, иконка, meta для iOS.
- Персонализация: `author` (имя + фото), `user.photo`, источник
  и теги у зикра, общий `comment` от автора.
- Личные заметки пользователя по каждому зикру с автосохранением
  (debounce 500мс + flush на навигацию). Иконка 📝 в списке.
- Кастомное число повторов с инлайн-пикером (✎). Метка `×N`
  в списке зикров. Автонапоминание «Готово» если уже накопил
  больше нового целевого.
- Трёхуровневое ручное обнуление прогресса: ↺ на зикре,
  «Обнулить» в шапке раздела, «Обнулить весь прогресс на
  сегодня» в ⚙.
- Меню настроек ⚙ с отдельными кнопками по типам данных
  (прогресс / заметки / кастомные повторы / полный сброс).
- Экспорт заметок в TXT через Blob.
- **Стабильные ID** у каждого зикра. Все пользовательские данные
  адресуются по `item.id`, перестановка/переименование в content.json
  безопасны.
- **Миграция localStorage** с (sectionId, idx) на zikrId: одноразовая,
  идемпотентная, с pre-backup и отчётом-плашкой пользователю.
- **Архив** осиротевших данных (зикр удалён из content.json — его
  заметки/повторы переезжают в архив, не теряются). Просмотр, скачивание
  TXT, удаление — через ⚙.
- **JSON-экспорт/импорт** всех пользовательских данных как страховка
  для переноса на другое устройство.
- **Сворачивание длинных текстов** в `comment` и личных заметках
  (граница 200 символов, обрезка до последнего пробела). User notes
  получают auto-resize в режиме редактирования.
- **Итерация 4 — разделение контента + публичный режим:**
  - Три слоя: `azkary.html` / `content.json` / `personal/*.json`.
  - URL `?u=name` выбирает учётку из `personal/`.
  - Два режима — `family` и `public`, единый код обоих сборок.
  - Защита публичной версии от подмены через `?u=`.
  - Экран ввода имени для публичной версии, «Изменить имя» в ⚙.
  - Два независимых блока комментариев на экране зикра — «Комментарий»
    (общий) и «💌 Тебе лично» (личный).
- **Итерация 5 — допиливание публичной версии:**
  - Предисловие автора (`letter`) вынесено **вверх** главного экрана,
    с разворачиванием длинного текста.
  - Экран **«Справка»** («Как пользоваться») — ссылка под предисловием
    и пункт в ⚙ Настройки.
  - Поле `feedbackEmail` + экран обратной связи «Предложи свой зикр
    или функцию» (адрес текстом, mailto, копирование).
  - Отдельный публичный проект `azkar-app` (см. README → деплой).
  - **Динамический manifest** (`setupDynamicManifest`): на старте
    статический `manifest.json` подменяется сгенерированным, у которого
    `start_url` сохраняет текущий `?u=`, а имя пользователя попадает в
    название ярлыка. Без этого установленное PWA семейной сборки
    запускалось из ярлыка **без** `?u=` → грузило отсутствующий
    `default.json` → экран ошибки. URL в манифесте абсолютные
    (относительные в blob-манифесте не резолвятся).
  - Фикс прокрутки: `render()` теперь всегда скроллит к началу страницы
    (`break` вместо `return` в `switch` — раньше `window.scrollTo` после
    него не срабатывал, и переход к следующему зикру оставлял прокрутку
    на середине).
  - Контент: добавлен саййидуль-истигфар (`morning-006`, Бухари 6306).
- **Итерация 6 — цветовые схемы:**
  - Выбор оформления в ⚙ → «Оформление»: `cream` (по умолчанию),
    `dawn`, `twilight`, `night`. Превью-плашки с реальными цветами.
  - Палитра вынесена в переменные CSS под `html[data-theme=…]`;
    единый источник истины по списку/цвету шапки/манифесту — массив
    `THEMES` в `azkary.html`.
  - Хранится в slug-независимом ключе `azkary.theme` (настройка
    устройства, общая для всех учёток браузера). Доступно в обоих
    режимах (`family` и `public`).
  - Применяется до первого `render()` (без мелькания фона); `<meta
    theme-color>` и фон PWA-манифеста синхронны выбранной схеме.
- **Итерация 7 — зона счёта (UX счётчика):**
  - Выбор активной зоны тапа на экране зикра (⚙ → «Счётчик»): `circle`
    (по умолчанию), `arabic`, `translit`, `translation`, `screen`.
  - Решает проблему длинных дуа: текст вверху, круг внизу — можно
    засчитывать повтор тапом прямо по читаемому блоку. Круг-табло
    остаётся всегда; у текстовой зоны — рамка-индикатор + бейдж-дубль.
  - Отличает тап от прокрутки (Pointer Events, допуск ~10px), исключает
    интерактивные элементы, сохраняет вибрацию.
  - Хранится в slug-независимом ключе `azkary.tapZone` (настройка
    устройства). Подробно — раздел «Экран зикра → Зона счёта».

**Очередь и будущие функции — в отдельном файле [`ROADMAP.md`](ROADMAP.md).**
Туда же переносятся пожелания, пришедшие на `feedbackEmail` с публички.
